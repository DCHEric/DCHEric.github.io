使用 Hexo + NexT + GitHub Pages 搭建个人博客：完整步骤与原理说明

发表于 2026-03-29 分类于 开源项目
详细记录我如何使用 Hexo、NexT 和 GitHub Pages 搭建个人博客，并解释每一步为什么这么做以及背后的技术原理。

这篇文章是我博客的第一篇正式内容。我想直接把这次博客从 0 到 1 的搭建过程完整记录下来：不仅写清楚怎么做，也写清楚为什么要这么做，以及这套方案背后的技术原理是什么。

我这次使用的方案是：

  • Hexo：静态博客生成器
  • NexT：Hexo 主题
  • GitHub Pages：静态网站托管服务

如果你也想低成本、可持续地搭一个个人博客，这套方案很合适。

一、先说结论：这套博客到底是怎么工作的

整个流程可以概括成一句话：

写 Markdown 文章 → Hexo 生成静态网页 → Git 推送到 GitHub → GitHub Pages 对外发布

也就是说，我平时真正编辑的内容并不是 HTML 页面本身，而是：

  • Markdown 文章
  • Hexo 根配置
  • NexT 主题配置
  • 图片等资源

然后交给 Hexo 统一生成最终网站。

二、为什么我选择 Hexo + NexT + GitHub Pages

这套方案吸引我的地方主要有几点：

  1. 成本低：GitHub Pages 可以免费托管静态网站。
  2. 写作简单：平时主要写 Markdown，不需要手写 HTML。
  3. 结构清晰：文章、页面、主题、配置分工明确。
  4. 适合长期维护：以后只要持续写文章和部署即可。
  5. 静态站点稳定：生成后就是 HTML/CSS/JS，部署简单，访问也稳定。

四、第二步：使用 Hexo 初始化博客

Hexo 是一个静态博客生成器。它的作用不是在线运行你的博客，而是把你的内容“编译”成静态网站文件。

Hexo 的核心思想很像这样：

  • 你写源文件
  • 它读取配置和主题
  • 它输出浏览器能直接访问的网页文件

在项目里最重要的几个目录分别是：

1. source/

这是站点内容目录，放你自己的内容：

  • source/_posts/：文章
  • source/about/：关于我页面
  • source/tags/：标签页
  • source/categories/：分类页
  • source/images/：你自己的图片

这是主题目录，放的是主题代码。

3. public/

这是最终生成目录。

Hexo 每次执行生成命令后，都会把最终网站输出到这里。这个目录中的文件才是实际部署到 GitHub Pages 的内容。

所以可以这样理解：

  • source/ 是原材料
  • themes/ 是模板
  • public/ 是成品

因为：

  • themes/ 是主题代码

十三、部署背后的原理到底是什么

很多人第一次接触时会觉得：执行一条命令，网站就上线了，好像很神奇。

其实拆开看并不复杂：

第一步：Hexo 把源内容转成静态文件

输入：

  • Markdown
  • 配置文件
  • 主题模板

输出：

  • HTML
  • CSS
  • JS
  • 图片资源

第二步：Hexo 通过 deploy 插件把 public/ 发布到 GitHub

这一步本质上是一次 Git 推送。

第三步：GitHub Pages 把这个仓库内容当成网站发布

所以整条链路其实就是：

写内容 → 生成网页 → 推送仓库 → GitHub 对外托管

这也是为什么静态博客部署起来会比传统后端网站简单很多：因为它根本不需要服务器运行数据库、接口或后端逻辑。

十四、为什么有时候“部署了却打不开”或者“没变化”

这类问题很常见，一般就几个原因：

1. url 或 root 配错

会导致：

  • CSS 丢失
  • 图片不显示
  • JS 路径错误

2. 没有重新 generate

你改了源文件，但 public/ 还是旧的。

3. 没有 deploy

本地改好了，但线上仓库还是旧内容。

4. GitHub Pages 还没生效

有时会有一点延迟。

5. 浏览器缓存

尤其是 CSS、图片和 JS，缓存后看起来会像“没更新”。
