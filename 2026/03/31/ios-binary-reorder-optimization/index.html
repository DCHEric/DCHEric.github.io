iOS二进制重排冷启动优化：原理、实操与反思

发表于 2026-03-31 分类于 iOS
深入探讨 iOS 二进制重排优化原理，从虚拟内存、Page Fault 到 Clang 插桩实战，以及为什么这项技术在现代 iOS 系统中已经过时。

iOS二进制重排冷启动优化

题外话

搞完才发现已经过时了。虽然没有有效结果，但是过程收获颇丰。

为什么二进制重排过时了

1. iOS 系统层面的优化

苹果在 iOS 13+ 对 dyld 做了大量优化：

  • dyld3 引入了启动闭包缓存，把符号绑定结果缓存起来，冷启动时直接读缓存
  • dyld4（iOS 15+）进一步优化，page fault 本身已经大幅减少

2. 收益越来越小

二进制重排针对的是 __TEXT 段的 page fault，但现在：

  • dyld 缓存覆盖了大部分系统库
  • 主二进制的 page fault 在整个启动耗时中占比已经很低
  • 抖音当年做重排收益是 15%，现在同样的工程可能只有 1-3%

3. 维护成本高

  • 每次大版本迭代都要重新收集 order 文件
  • Swift flags 兼容性问题
  • 收益越来越小但维护成本不变

现在更值得做的启动优化

  1. 动态库合并：减少 dyld 加载动态库的数量
  2. +load 方法治理：延迟非必要的初始化
  3. 启动阶段懒加载：首屏之外的功能延后初始化
  4. 预编译优化：减少首次启动的编译时间

一、什么是二进制重排

（一）APP启动过程

要想优化APP的启动速度，首先得要知道APP的启动流程是啥？

iOS APP启动流程可以分为以下几个阶段：

详细流程说明

1. 内核加载阶段（Pre-main，系统负责）

  • 用户点击图标，系统调用 fork() 创建新进程
  • 内核加载 Mach-O 可执行文件到内存
  • 检查 Mach-O 文件签名和权限
  • 为进程分配虚拟内存空间

2. dyld（动态链接器）阶段（Pre-main，耗时大头）

  • 加载所有依赖的动态库（Framework、dylib）
  • 递归加载动态库的依赖
  • Rebase：修正内部指针地址（ASLR地址随机化）
  • Bind：绑定外部符号（函数、变量）
  • Objc setup：注册所有 Objective-C 类、分类、方法
  • Initializers：执行 C++ 静态构造函数、Attribute((constructor))

3. Runtime 初始化阶段（Pre-main）

  • 执行所有类的 +load 方法（按照依赖顺序）
  • Method Swizzling 黑魔法通常在这里执行
  • 注意：+load 是阻塞主线程的，耗时操作会明显拖慢启动

4. main() 函数（Post-main 开始）

5. UIApplicationMain 阶段

  • 创建 UIApplication 单例对象
  • 创建 AppDelegate 实例
  • 开启主线程 RunLoop
  • 创建 UIWindow

6. AppDelegate 生命周期

启动时间划分

阶段 耗时占比 优化方向
Pre-main 20%-40% 减少动态库数量、治理+load方法
main到首屏 60%-80% 懒加载、异步初始化、减少首屏业务

二进制重排优化的就是 dyld 阶段的 Page Fault，通过调整函数在 Mach-O 文件中的顺序，减少启动时的缺页中断次数。

（二）虚拟内存

首先需要知道为什么要有虚拟内存的存在？

如果没有虚拟内存，程序直接操作物理内存，会有几个严重的问题：

1. 程序之间相互踩踏

微信的代码如果有 bug，直接写了 0x1500 这个地址，就把抖音的数据覆盖了。程序之间没有任何隔离，一个 bug 能搞崩整个系统。

2. 内存不够用时直接没法运行

物理内存只有 6GB，但如果同时跑：

没有虚拟内存的话，直接报错，程序跑不起来。

3. 程序加载地址冲突

每个程序编译时都假设自己从某个固定地址开始，两个程序都想从 0x1000 开始，没法同时运行。

虚拟内存能解决什么问题？

每个进程都以为自己独占全部内存，但实际上是假的：

中间有一个映射表（页表）负责翻译：

iOS/macOS 的页大小：16KB（ARM64）

可以带来几个好处：

  • 隔离性：同样的虚拟地址，映射到不同物理地址。进程 A 永远访问不到进程 B 的内存，天然隔离。
  • 懒分配：程序申请了内存，不代表立刻占用物理内存。程序申请的内存远大于实际用到的内存，系统按需分配。
  • Swap：物理内存不够时，把暂时不用的内存页存到磁盘。macOS 有 Swap，所以 Mac 可以开很多程序。iOS 没有 Swap（闪存寿命），内存不够直接杀进程。

（三）Page Fault问题的出现

启动优化的二进制重排，就是为了减少情况3的发生次数——把启动时用到的代码集中在少数几页里，减少 Page Fault。

App 的二进制文件躺在磁盘上，启动时不是一次性全部读入内存的，而是按页懒加载：

每访问一个新的页，就触发一次 Page Fault，系统要：

单次 Page Fault 大约耗时 0.1ms ~ 1ms，启动时可能触发几百次，累计就是几百毫秒。

（四）二进制重排详解

核心思路：把启动时用到的函数，集中排列在二进制文件的前几页

实际项目里启动函数可能分散在几百页，重排后集中到几十页，效果非常显著。

二、Lisa冷启动page fault现状

通过Instruments工具集中的System Trace工具，可以看到在冷启动的时候，”File Backend Page in”的数量为10671，这就是缺页中断Page Fault的数量，整体启动耗时833.31ms

三、优化实操

实际上，二进制重排的步骤并不复杂，真正的难点在于如何按照函数的执行顺序去重新排列页表中的page。

Xcode使用的链接器是 ld，ld中有一个参数是 order file，order file是一个文件路径，它指向了order文件，order文件中写入的是符号的顺序，Xcode在编译打包的时候就会生成按照order文件中的符号顺序排列的可执行文件。

（一）查看二进制可执行文件中的符号顺序

如图所示，当前项目工程里面的所有的源文件都是记录在Compile Sources里面。每一个源文件在编译的时候都会生成一个目标文件（.o），然后将所有的.o以及静态库等链接成一个Mach-O，这个链接的顺序就是按照Compile Sources里面的顺序来的，而这里的顺序是可以手动拖动的。

所以说，文件的顺序就确定了。

（二）LinkMap 文件里有什么

1. 目标文件列表

这里面记录的是所有参与链接的 .o 文件，文件顺序就是Compile Sources里面的顺序。

2. 段和区域布局

记录的是MachO二进制可执行文件里面段的一些数据，Segment这一列表示是哪一段：

3. 符号表（最有用）

每个函数在二进制里的地址、大小、来自哪个文件。

Symbols里面的数据有四列：Address、Size、File、Name

  • Name：方法名或者函数名
  • File：在哪一个文件当中，这里面的数字给最上方Object files里面的数字是对应的
  • Size：这个方法或者函数占用的空间大小，函数里面的内容多少不一样，其Size也是不一样的，100行代码的方法肯定比1行代码的方法的Size要大。
  • Address：这个方法或者函数的真实的地址，不是这个方法对应的符号地址（符号地址就是存储在MachO文件的Data段中的符号）。二进制重排，实际上就是将相关代码的所有内容放到前面去，而不仅仅是简简单单将符号放到前面。

LinkMap 文件的作用：

  • 分析包大小：哪个类、哪个库占了多少空间，一目了然，做包体积优化必备。
  • 验证二进制重排：对比 order file 配置前后，函数排列顺序是否真的改变了。
  • 分析符号顺序：启动优化时，确认启动相关函数是否都集中在前面几页。
  • 排查符号冲突：链接报错时，找到哪两个库定义了同名符号。

无论是方法还是函数还是block，它们在调用的时候，首先都会调用 __sanitizer_cov_trace_pc_guard 函数。也就是说，当配置了Clang代码插入工具之后，编译器会在编译的时候在所有的方法、函数、block内部都加入了一条调用 __sanitizer_cov_trace_pc_guard 函数的汇编代码，这就是所谓的Clang静态插桩，Hook一切。

作用：被动收集。编译器插桩后，每当任意函数被调用，这里就自动记录一次，把函数地址塞进 symbolList 队列里。你不需要主动调用它，它是自动触发的。

反思

虽然这次实验结果并不理想（甚至变慢了），但通过这个过程，我深入理解了：

  • 虚拟内存和 Page Fault 的底层原理
  • Mach-O 文件结构和符号链接过程
  • Clang 插桩技术的实现机制
  • LinkMap 文件的作用和分析方法

更重要的是，我意识到盲目追求”业界成熟方案”并不可取，需要根据实际情况（iOS版本、项目规模、启动流程）判断优化手段是否适用。有时候，”不做优化”也是一种正确的决策。
